# email-drafting

## Overview

クライアントへのビジネスメールを状況に応じて作成するスキル。

---

## 🔥 プロフェッショナル・スタンス（最重要）

### 1. 「できない」と言わない

ビジネスにおいて、単に「No」と答えることは思考停止と同義です。
常に**「解決策（代替案）」**を提示し、**「主導権（ボール）」**を自社で持ち続ける。

```
❌ 思考停止の回答:
「あいにくスケジュールが埋まっており、参加できません。」

✅ 解決志向の回答:
「あいにくその日時は先約がございますが、翌日の午後であれば調整可能です。いかがでしょうか？」
```

### 2. 相手に仕事をさせない

相手にボール（タスク）を持たせたままにすることは、プロジェクトの遅延を意味する。

### 3. 自責タスク最大化の法則（7:3のルール）

議事録やフォローアップメールにおいて、**相手のタスクよりも自社のタスクを多く記載**する。

```
❌ 相手任せ:
- クライアント様：見積もりの承認

✅ 自社主導:
- クライアント様：見積もりの承認
- 自社：承認用比較データの送付（明日中）
- 自社：承認後のスケジュール案の作成
```

## 📧 メール構造の鉄則

### 宛先・件名のエンジニアリング

**To（宛先）**:
- ボール（責任）を持っている人を指定
- 原則1名、あるいは本文で指名する
- 「誰かやってくれるだろう」を防ぐ

**件名（Subject）**:
開封の可否を決める「見出し」。結論・目的・期限を明確に。

```
❌ NG: 「ご相談」「例の件」「お願い」
✅ 推奨: 【回答期限：1/25】Aプロジェクト見積もりの件（再送）
✅ 推奨: 【ご確認】〇〇施設様 契約書送付の件
✅ 推奨: 【お礼】1/22 オンラインMTGについて
```

**Bccの原則禁止**:
Bccは「事故の元」。使用時はダブルチェック必須。

### 本文の構成（PREP法）

スマホで見る相手を想定し、**結論ファースト**を徹底。

1. **結論・目的**（何をしてほしいのか、何が決まったのか）
2. **詳細・理由**（箇条書きを活用）
3. **結び**（次のアクション、締めの挨拶）

## Email Types

### 1. フォローアップメール（MTG後）

```yaml
trigger: 商談後、提案後、MTG後
tone: professional, warm
deadline: 当日中（遅くとも翌日午前中）
structure:
  - お礼
  - 決定事項の確認
  - ToDo / Next Action（自社タスクを多めに）
  - 保留事項（あれば）
  - 次回予定
```

**重要**: 議事録メールは単なる記録ではなく、プロジェクトを動かすための**「契約書」**。

### 2. アポイント依頼メール

```yaml
trigger: 新規アプローチ、再アプローチ
tone: professional, concise
structure:
  - 自己紹介（必要に応じて）
  - 目的（相手のメリットを先に）
  - 候補日時（3つ以上）
  - 調整の柔軟性を示す
```

### 3. 提案メール

```yaml
trigger: 提案送付
tone: professional, compelling
structure:
  - 課題の確認（相手の言葉を引用）
  - 提案概要
  - ベネフィット（数字で示す）
  - 次のステップ（自社アクションを明記）
```

### 4. リマインダーメール

```yaml
trigger: 返信なし、期日前
tone: professional, gentle
structure:
  - リマインド（責めない）
  - 前回の要点（簡潔に）
  - 代替案の提示（相手が迷っている場合）
  - 再度のお願い
```

### 5. 断りへの対応メール

```yaml
trigger: 相手からの断り、難色
tone: professional, solution-oriented
structure:
  - 理解を示す
  - 代替案A（条件変更）
  - 代替案B（スコープ変更）
  - 再提案の余地を残す
```

## 上級テクニック

### 1. インライン返信の作法

複数の質問に答える場合、相手のメールを引用し、質問の下に回答を書く。

**マナー**: 冒頭に「**※行間への書き込みにて失礼いたします**」と添える。

```
※行間への書き込みにて失礼いたします。

> 1. 納期について教えてください。
→ 2月末を予定しております。

> 2. 費用の内訳はありますか？
→ 添付の資料をご確認ください。
```

### 2. 件名の「Re:」は消さない

**理由**: メーラーのスレッド機能が解除され、過去の経緯が追えなくなる。
**例外**: 話のテーマが完全に変わる場合のみ、新規で件名を設定。

### 3. 添付ファイルのルール（脱PPAP）

パスワード付きZipは多くの企業で**受信拒否**される。

- **3MB未満**: そのまま添付
- **3MB以上**: クラウドストレージ（Google Drive等）のリンクを発行

## 架電タイミング（宿泊施設向け）

宿泊施設への電話は、以下のタイミングを厳守：

| 避けるべき時間帯 | 理由 |
|----------------|------|
| **10:00〜11:00** | チェックアウト対応 |
| **15:00〜17:00** | チェックイン対応 |

| 狙い目の時間帯 | 理由 |
|---------------|------|
| **12:00〜14:00** | フロント業務の谷間 |
| **11:30〜12:00** | チェックアウト後の落ち着き |

## Quality Checklist

### 送信前チェック（必須）

- [ ] 宛先は正しいか（Toは1名が原則）
- [ ] 件名は期限・目的が明確か
- [ ] 結論ファーストになっているか
- [ ] 自社タスクが相手タスクより多いか（7:3）
- [ ] 「できない」と言っていないか（代替案はあるか）
- [ ] 誤字脱字はないか
- [ ] 署名は入っているか
- [ ] CCに tokidakohei@reiwatravel.co.jp が入っているか
- [ ] 添付ファイルは正しいか（該当時）

## Best Practices

### DO ✅

- ✅ 結論ファースト
- ✅ 代替案を常に用意
- ✅ 自社タスクを多く持つ
- ✅ 期限を件名に入れる
- ✅ 相手の負担を減らす

### DON'T ❌

- ❌ 「できません」で終わる
- ❌ 相手にボールを投げっぱなし
- ❌ 曖昧な件名
- ❌ 長すぎるメール
- ❌ パスワード付きZip
