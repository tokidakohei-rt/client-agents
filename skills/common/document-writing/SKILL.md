# document-writing

## Overview

ビジネス文書、レポート、分析結果などを構造化して作成するスキル。

---

## Principles

### 1. BLUF (Bottom Line Up Front)

**結論を最初に書く**

```markdown
❌ Bad:
「様々な要因を分析した結果...（長い説明）...よってAを推奨します」

✅ Good:
「推奨: A案を採用
理由: コスト効率が最も高く、リスクが許容範囲内のため」
```

### 2. 30秒ルール

**最初の30秒で要点が伝わる**

```markdown
## サマリー
- 結論: {1行}
- 根拠: {2-3行}
- 次のアクション: {1行}
```

### 3. Pyramid Principle

```
結論
├── 根拠1
│   ├── 詳細1-1
│   └── 詳細1-2
└── 根拠2
    ├── 詳細2-1
    └── 詳細2-2
```

### 4. MECE

**漏れなく、ダブりなく**

## Document Types

### 1. 分析レポート

```markdown
# {タイトル}

## サマリー
{結論と主要な発見を3行以内で}

## 分析結果
### {分析項目1}
{データ、図表、考察}

### {分析項目2}
{データ、図表、考察}

## 結論
{分析結果のまとめ}

## 推奨アクション
1. {アクション1}
2. {アクション2}
```

### 2. ステータスレポート

```markdown
# {クライアント名} ステータスレポート

## 基本情報
| 項目 | 内容 |
|------|------|
| 現在ステージ | {STAGE} |
| ヘルススコア | {SCORE}/100 |
| 次回アクション | {ACTION} |

## 直近の活動
- {DATE}: {ACTIVITY}

## 課題・リスク
- {ISSUE}

## 推奨アクション
1. {ACTION} - 期限: {DEADLINE}
```

### 3. 日次/週次サマリー

```markdown
# {期間} サマリー

## ハイライト
- {HIGHLIGHT_1}
- {HIGHLIGHT_2}

## 数値サマリー
| 指標 | 今期 | 前期 | 変化 |
|------|------|------|------|
| {KPI} | {N} | {N} | {+/-X}% |

## アクションアイテム
- [ ] {ACTION_1} - 担当: {OWNER}
- [ ] {ACTION_2} - 担当: {OWNER}

## 来週の予定
- {PLAN}
```

## Formatting Standards

### 見出し階層

```markdown
# H1: 文書タイトル（1つのみ）
## H2: 主要セクション
### H3: サブセクション
#### H4: 詳細（必要時のみ）
```

### 箇条書き

```markdown
- 並列な項目
  - サブ項目
- 次の項目
```

### 表

```markdown
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| データ | データ | データ |
```

### 強調

```markdown
**重要**: 太字で強調
*補足*: イタリックで補足
`コード`: バッククォートで技術用語
```

## Quality Checklist

```markdown
## 文書品質チェック

### 構造
- [ ] BLUF（結論が最初）
- [ ] 30秒ルール（要点がすぐわかる）
- [ ] 見出しで構造化
- [ ] MECE（漏れなくダブりなく）

### 内容
- [ ] 具体的な数値
- [ ] 出典明記（該当時）
- [ ] 事実と推測の分離
- [ ] 前提・制約の明示

### アクション
- [ ] 次のアクションが明確
- [ ] 担当者・期限あり（該当時）
- [ ] 優先度が明確

### 可読性
- [ ] 適切な見出し階層
- [ ] 箇条書き活用
- [ ] 表・図の活用
- [ ] 適切な強調
```

## Best Practices

### DO ✅

- ✅ 結論を最初に書く
- ✅ 30秒で要点が伝わる
- ✅ 見出しで構造化
- ✅ 数値を具体的に
- ✅ アクションアイテムに担当・期限

### DON'T ❌

- ❌ 長い段落
- ❌ 曖昧な表現（「大きい」「多い」等）
- ❌ 結論が最後
- ❌ 担当・期限なしのアクション
